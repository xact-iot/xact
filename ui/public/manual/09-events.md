# Events

Events are the system's audit trail and alarm log. Every significant occurrence - alarm activations, user actions, device status changes, and system messages - is recorded as an event and can be viewed, filtered, and exported.

## Event Attributes

Each event record contains:

| Field | Description |
|-------|-------------|
| **Timestamp** | When the event occurred (RFC3339 format) |
| **Severity** | The importance level of the event |
| **Device** | The device associated with the event (if applicable) |
| **User** | The user who triggered the event (if applicable) |
| **Message** | A human-readable description of what happened |
| **Parameters** | Additional structured data (optional, depends on event type) |
| **Notification ID** | Links to a notification profile for alert delivery (if applicable) |

## Severity Levels

Events are classified by severity, from least to most critical:

| Level | Use | Example |
|-------|-----|---------|
| **DEBUG** | Diagnostic information for troubleshooting | Pipeline debug output, configuration changes |
| **INFO** | Normal operational events | User login, device connected, report generated |
| **WARN** | Conditions that may require attention | Device communication delay, approaching storage limit |
| **ERROR** | Failures that affect functionality | Failed to send notification, database write error |
| **CRITICAL** | Severe failures requiring immediate action | Database connection lost, MQTT broker unavailable |

## How Events Are Generated

Events come from several sources:

### Processing Block Events

The **Limit Check** and **Flap Filter** processing blocks generate events when alarm conditions change:

- **Alarm raised** - when a tag value exceeds a configured limit
- **Alarm returned** - when a tag value returns to normal range
- **Flapping detected** - when rapid alarm oscillation is detected
- **Flapping cleared** - when the alarm state stabilises

Each of these events can be configured with a severity level and custom message in the processing block parameters. They can also be linked to notification profiles for automatic alert delivery.

### System Events

XACT generates events for significant system operations:

- User authentication (login, logout, failed login attempts)
- Configuration changes (dashboard edits, permission updates)
- Data ingestion events (new device connected, ingest errors)

### Manual Events

Events can also be generated by external integrations via the REST API.

## Events Viewer Widget

The **Events Viewer** widget (System category) provides a full-featured interface for browsing and analysing events.

### Filtering

Narrow down the event list using:

- **Time range** - select a preset period (1h, 3h, 6h, 12h, 24h) or a custom range
- **Severity** - show only events at or above a selected severity level
- **Device** - filter to events from a specific device
- **Search** - free-text search across all event fields

### Sorting

Click any column header to sort the event list. Click again to reverse the sort direction. The default sort is by timestamp (newest first).

### Pagination

For large result sets, events are paginated. Configure the page size and navigate between pages using the controls at the bottom of the widget.

### Auto-Refresh

The widget can automatically poll for new events at a configurable interval:

- **Off** - manual refresh only
- **10 seconds** - frequent updates for active monitoring
- **30 seconds**, **1 minute**, **5 minutes** - less frequent updates

The widget uses incremental loading - on each refresh, it only fetches events newer than the last received event, rather than reloading the entire dataset.

### CSV Export

Click the **Export** button to download the current filtered event set as a CSV file. This is useful for offline analysis, incident reports, or archiving. The export uses SheetJS for reliable spreadsheet-compatible formatting.

## Events and Notifications

Events with a **Notification ID** are linked to the notification system. When such an event is generated, the corresponding notification profile determines which users are alerted and through which channels (email, Telegram). See the [Notifications](#notifications) chapter for details on configuring notification profiles.

## Event Retention

Event records are stored in the database and are subject to the same retention policies as other log data. Old events are periodically purged based on the configured retention period.
